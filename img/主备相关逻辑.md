### 相关的控制操作

门禁：BackUp_IsLogicMaster() 这个是用于判断是否为主的相关函数 只有主才做业务

**1) 主备数据同步（运行期同步）**

核心条件：只有**逻辑主**才做同步。

只有逻辑主会驱动运行期同步；逻辑备主要接受数据。

同步开关：BackUp_IsAllowSync() / BackUp_SetSyncSwitch

BackUp_SyncRunDataTask（运行数据同步任务）



**2) 逻辑主将程序/数据文件同步到逻辑备（升级同步文件）**

逻辑主通过 ASLink 请求备机拉取数据/程序文件，备机侧实际下载文件。

入口在 WNM 升级命令（WNM_HandlerSysUpgrade）发起后：

主机发送升级命令 → BackUp 收到 → 若需要备机参与，发送 ASLink 升级请求
BackUp_RxWnmMsg → BackUp_ASLinkTxUpgradeReq

备机拉取文件：

BackUp_DownLoadFile(EN_BACKUP_SYNC_DATA/PROGRAM)



**3) 逻辑备加载新程序（升级生效）**

在备机端，升级任务触发 LOAD_Transfer：BackUp_UpgradeTask

具体加载：
LOAD_Transfer → LOAD_CheckProgram → LOAD_SaveProgram

备机完成程序写盘，但不会立即切换运行，需要复启。



**4) 复启逻辑备机，使其激活新程序**

升级流程中会触发复位/软复位，或通过管理面切换触发重启：

备机重启（示例）：BackUp_ASLinkRxRebootPeer → DEV_SystemReset(3)

重启后程序切换：

LOADCTRL_SetNextRunProgram 只写 nextprogram

Daemon 在重启后读取 nextprogram 启动新程序

稳定后固化 currentprogram/nextprogram

复启是生效的关键，Daemon 会按 nextprogram 启动新版本。



**5) 主备切换（业务接管）**

切换由 **Mas/Sla 协商** 和 **逻辑主门禁**完成：

协商流程：BackUp_ASLinkTxMasSlaReq → BackUp_ASLinkRxMasSlaReq → BackUp_ASLinkRxMasSlaResp

逻辑主最终设置：BackUp_SetLogicMaster(bIsMaster)

业务线程门禁：大量线程只在 BackUp_IsLogicMaster() 为真时处理业务（如 [AMF_thread.c](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/lantian/.vscode/extensions/openai.chatgpt-0.4.71-win32-x64/webview/#) 等）

切换后逻辑主接管业务，逻辑备暂停业务处理。

配置/etc/keepalived/keepalive.conf

主要配置点在抢占的逻辑 
	这里例子的意思:priority 优先级 数字越大，权重越高。在选举 Master 时，数字大的优先。state BACKUP配合nopreempt使用，意思为如果这台机器（优先级 102）坏了，VIP 漂移到了备份机（优先级 100）；当这台机器修好重启后，不会强行把 VIP 从备份机那里抢回来。这通常需要先启动主再启动备，谁先启动则为谁做业务，也可以通过后续的主备切#换来指定切换谁来做业务(其实这个操作本质也是重启而已)。virtual_ipaddress { 10.9.226.27 }：定义的虚拟IP(VIP)。业务请求应该访问这个IP。authentication：auth_type PASS：使用明文密码认证。auth_pass 1111：认证#密码。主备必须一致，防止同一网段内其他 Keepalived 实例干扰。interface enp3s0：绑定 VIP 的物理网卡名称。#Keepalived 会在这个网卡上监控流量和发送心跳。virtual_router_id 51：虚拟路由 ID。主、备两台机器的这个 ID #必须完全一致，否则它们会认为自己属于不同的集群。

​	**若想主恢复后回到主上做业务则主需要配置state MASTER 备 state BACKUP** 
​	**不能添加nopreempt  priority必须 主的值大于备**

## 升级主备操作的返回值意义

get操作需要判断的返回值 一共有两个 upGradeCmdType 和upGradeCmdResult 

#### upGradeCmdType 表示为当前操作的类型 

返回的值的意思为“当前/最近一次升级相关操作的类型”



#### upGradeCmdResult 表示对应操作当前的状态

EN_BACKUP_UPGRAGE_BUTT(255)   **表示处理中/暂无结果**

EN_BACKUP_UPGRAGE_FAILED(1)  **超时**（20 分钟未结束）

EN_BACKUP_UPGRAGE_SUCCESS(0)  **处理成功**

EN_BACKUP_UPGRAGE_FAILED(1) **处理失败**



#### 具体例子

**备板复位返回值** 

通常发了一个Action为EN_EMWEB_BACKUP_RESET(5) 的操作 返回值大致有如下的几种：

如果为upGradeCmdType=EN_EMWEB_BACKUP_RESET(5)     upGradeCmdResult=EN_BACKUP_UPGRAGE_BUTT(255)  正在处理 则证明正在处理该操作

如果为upGradeCmdType=EN_EMWEB_BACKUP_RESET(5)      upGradeCmdResult=EN_BACKUP_UPGRAGE_SUCCESS(0) 成功

则证明该操作完成了并且是成功的

如果为upGradeCmdType=EN_EMWEB_BACKUP_RESET(5)       upGradeCmdResult=EN_BACKUP_UPGRAGE_FAILED(1)失败



也可能为其他的值：
```
    EN_BACKUP_UPGRAGE_CMD_TYPE              =2,             //命令类型错误
    EN_BACKUP_UPGRAGE_FILE_ERROR            =4,	            //文件处理失败
    EN_BACKUP_UPGRAGE_NO_SLAVE_BOARD        =5,	            //备板不存在
    EN_BACKUP_UPGRAGE_FAULE_SLAVE_BOARD     =6,	            //备板故障
    EN_BACKUP_UPGRAGE_SLAVEBOARD_NET_FAULT  =7,             //备板网口状态故障
    EN_BACKUP_UPGRAGE_SLAVEBOARD_COMM_FAULT =8,             //备板通讯故障
    EN_BACKUP_UPGRAGE_FILE_NOT_EXIST        =9,             //文件不存在
```



**注意数据同步开个设置这两个比较特殊只有设置成功以及设置失败两个返回值**

upGradeCmdType=EN_EMWEB_OPEN_SYNC_DATA_BACKUP(7 )  upGradeCmdResult=EN_BACKUP_UPGRAGE_SUCCESS(0) 成功



upGradeCmdType=EN_EMWEB_OPEN_SYNC_DATA_BACKUP(7 )

upGradeCmdResult=EN_BACKUP_UPGRAGE_FAILED(1)失败



GradeCmdType=EN_EMWEB_CLOSE_SYNC_DATA_BACKUP(8 )  upGradeCmdResult=EN_BACKUP_UPGRAGE_SUCCESS(0)成功



upGradeCmdType=EN_EMWEB_CLOSE_SYNC_DATA_BACKUP(8 ) upGradeCmdResult=EN_BACKUP_UPGRAGE_FAILED(1)失败



​	一句话总结就是：upGradeCmdType 回应在做哪种升级操作；upGradeCmdResult 回应这个操作当前的结果/状态（成功、失败、处理中等）。如果是处理中说明该操作还没完成最好不要做其他的操作，建议当返回值为0也就是 upGradeCmdResult==EN_BACKUP_UPGRAGE_SUCCESS(0) 再进行下一个操作。



升级的大致步骤

升级备机

先将升级文件放到逻辑主指定的目录下 使用同步文件操作，当返回值为成功时，再使用升级备机操作，当返回值为成功时，再使用备机重启操作，返回值为成功时证明备机升级完成。
